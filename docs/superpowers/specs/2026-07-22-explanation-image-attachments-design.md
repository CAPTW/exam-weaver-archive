# 해설 이미지 첨부 설계

## 목적

문항의 텍스트 해설에 이미지 한 장을 첨부할 수 있게 한다. 사용자는 문제 관리
우측 해설 패널과 문제 수정창에서 이미지 파일을 선택하거나 클립보드 이미지를
붙여넣고 삭제할 수 있다. 문제 풀이 화면은 저장된 해설 이미지와 텍스트를 읽기
전용으로 표시한다.

이번 단계는 실제 사용성을 먼저 검증하기 위한 1단계다. 문항당 이미지 한 장만
허용하지만, 저장 구조는 이후 여러 이미지와 본문 내 삽입 위치를 지원할 수 있게
설계한다. DOCX 출력, 여러 이미지 편집, 본문 내 인라인 배치는 이번 범위에서
제외한다.

## 데이터 모델

`question_explanation_images` 테이블을 추가한다.

| 열 | 의미 |
|---|---|
| `id` | `INTEGER PRIMARY KEY AUTOINCREMENT` 첨부 이미지 식별자 |
| `question_id` | `questions(id)`를 참조하는 필수 값. 문항 삭제 시 함께 삭제 |
| `image_path` | 앱 관리 이미지 파일의 필수 상대 경로 |
| `display_order` | 필수 이미지 순서. 1단계에서는 항상 `0` |
| `alt_text` | 향후 접근성과 인라인 이미지 설명에 사용할 nullable 텍스트 |
| `created_at` | 기본값이 현재 시각인 생성 시각 |

`(question_id, display_order)`에는 고유 제약을 둔다. 1단계 Repository는 문항당
`display_order = 0`인 행을 최대 하나만 생성한다. 향후 2단계는 동일 테이블에
순서가 다른 행을 추가하고, 별도 본문 문서 모델이 이미지 `id`를 참조하게 한다.
따라서 1단계 데이터를 다른 테이블로 옮길 필요가 없다.

기존 DB는 `ExamRepository.init_database()`의 멱등 마이그레이션으로 새 테이블과
인덱스를 생성한다. 문제 삭제는 외래 키 `ON DELETE CASCADE`로 첨부 행을 제거한다.

## 관리 이미지 저장소

선택하거나 붙여넣은 이미지는 `data/explanation_images` 아래에 UUID 기반 파일명으로
복사한다. 파일 선택은 PNG, JPG, JPEG를 허용하고 Qt 이미지 디코더로 실제 이미지인지
검증한다. 클립보드 이미지는 PNG로 저장한다. DB에는 앱 기준 상대 경로
`data/explanation_images/<uuid>.<ext>`를 기록하고, 화면 표시와 파일 정리 시
`get_base_dir()`를 기준으로 절대 경로를 해석한다.

외부 원본 파일은 읽기만 하며 이동하거나 삭제하지 않는다. 이미지 교체 또는 삭제 시
앱 관리 폴더 안에 있고 다른 첨부 행이 참조하지 않는 파일만 정리한다. 관리 폴더 밖의
경로는 어떤 경우에도 삭제하지 않는다.

## Repository와 트랜잭션

Repository는 문항 조회 결과에 `explanation_images` 목록을 포함한다. 각 항목에는
최소한 `id`, `image_path`, `display_order`, `alt_text`가 들어간다. 단일 DB와
`MountedExamRepository`의 namespaced 문항 조회 모두 같은 형태를 반환한다.

해설 저장 API는 해설 텍스트와 1단계 이미지 상태를 한 트랜잭션에서 갱신한다.

1. UI가 선택 또는 붙여넣기한 이미지를 임시 상태로 보관한다.
2. 저장 시 검증된 이미지를 관리 폴더에 기록한다.
3. 같은 트랜잭션에서 `questions.explanation`과 첨부 이미지 행을 갱신한다.
4. DB 저장이 실패하면 새 관리 파일을 제거하고 기존 DB 행과 기존 파일을 유지한다.
5. DB 저장이 성공한 뒤 교체되거나 삭제된 이전 관리 파일을 참조 여부 확인 후 정리한다.

문항 삭제 전에는 연결된 관리 파일 경로를 수집한다. 문항과 첨부 행 삭제가 커밋된 뒤,
다른 첨부 행이 참조하지 않는 관리 폴더 내부 파일만 제거한다. DB 롤백 시에는 파일을
제거하지 않는다.

쓰기 가능한 마운트는 namespaced 문항 ID를 해당 로컬 Repository로 위임한다. 읽기 전용
마운트는 이미지를 표시할 수 있지만 저장 시 기존 쓰기 차단 오류를 사용자에게 보여준다.

## 공용 해설 이미지 편집 영역

문제 관리와 문제 수정창이 같은 동작을 사용하도록 공용 해설 이미지 편집 위젯을 둔다.
위젯은 다음 책임만 가진다.

- 현재 저장 이미지 또는 저장 전 후보 이미지 미리보기
- `이미지 선택`, `붙여넣기`, `삭제` 동작
- 이미지 경로와 대기 중인 변경 상태 관리
- 이미지 비율을 유지한 패널 너비 맞춤 미리보기
- 이미지 없음, 붙여넣은 이미지, 파일 누락, 읽기 실패 상태 표시

선택 또는 붙여넣기는 미리보기만 바꾸며 즉시 DB에 쓰지 않는다. 문제 관리에서는
`저장`, 문제 수정창에서는 수정 완료 시점에 해설 텍스트와 함께 반영한다. 기존 `비우기`
버튼은 텍스트만 지우고 이미지 제거는 명시적인 `삭제` 버튼으로만 수행한다.

## 화면별 동작

### 문제 관리 우측 해설 패널

해설 입력창 아래에 공용 이미지 편집 영역을 배치한다. 문항의 해설을 열면 텍스트와
첫 번째 첨부 이미지를 함께 로드한다. 저장 성공 후 현재 상태를 다시 로드하고, 실패하면
사용자가 편집 중인 텍스트와 이미지 후보를 유지한다.

### 문제 수정창

기존 해설 사이드카의 텍스트 입력창 아래에 같은 이미지 편집 영역을 배치한다.
`get_data()`가 해설 이미지 변경 상태를 반환하고 Repository가 문제의 다른 필드와 함께
저장한다. 문제 복제 시 원본 해설 이미지는 같은 경로를 공유하지 않고 새 관리 파일로
복사해 독립된 첨부로 만든다.

### 문제 풀이

정답 공개 조건을 충족하고 사용자가 `해설 보기`를 펼쳤을 때 해설 텍스트 아래에 첫 번째
이미지를 표시한다. 이미지는 원래 비율을 유지하면서 풀이 영역 너비와 지정한 최대 높이
안에 맞춘다. 해설 텍스트 또는 이미지 중 하나만 있어도 해설 보기 버튼을 제공한다.

## 오류 처리

- 클립보드에 이미지가 없으면 현재 후보 이미지를 바꾸지 않고 상태 안내를 표시한다.
- 확장자가 허용돼도 Qt가 디코딩하지 못하는 파일은 복사하거나 저장하지 않는다.
- 관리 폴더 생성, 파일 복사 또는 PNG 저장 실패는 사용자에게 오류로 알린다.
- DB 저장 실패 시 기존 해설과 이미지의 영속 상태를 유지한다.
- DB 경로에 해당하는 파일이 없거나 손상됐으면 텍스트 해설은 정상 표시하고 이미지
  영역에는 `이미지 파일 없음` 또는 `이미지를 읽을 수 없음`을 표시한다.
- 파일 정리는 해석된 절대 경로가 관리 폴더 내부인지 확인한 뒤 수행한다.

## 테스트 전략

### Repository

- 기존 DB 초기화 시 새 테이블과 인덱스가 생성된다.
- 해설 텍스트와 이미지 추가, 교체, 삭제가 함께 저장된다.
- 문제 삭제 시 첨부 행이 제거된다.
- DB 실패 시 기존 행이 유지되고 새 관리 파일이 정리된다.
- 다른 행이 참조하는 파일과 관리 폴더 밖의 파일은 삭제하지 않는다.
- 단일 DB와 마운트 Repository가 같은 첨부 목록 형태를 반환한다.
- 쓰기 가능 마운트는 올바른 원본 DB를 갱신하고 읽기 전용 마운트는 거부한다.

### 이미지 저장소와 UI

- 파일 선택 이미지를 UUID 이름으로 관리 폴더에 복사한다.
- 클립보드 이미지를 PNG로 저장한다.
- 이미지가 없는 클립보드와 손상된 파일은 기존 상태를 바꾸지 않는다.
- 문제 관리와 문제 수정창에서 추가, 붙여넣기, 삭제 후 미리보기가 갱신된다.
- 저장 후 화면을 다시 열어도 텍스트와 이미지가 유지된다.
- 문제 풀이에서 텍스트만, 이미지만, 텍스트와 이미지가 함께 있는 해설을 표시한다.
- 누락되거나 손상된 이미지가 있어도 각 화면이 중단되지 않는다.

기존 텍스트 전용 해설, 문제 이미지, 선지 이미지, 문제 복제, 마운트 쓰기 라우팅에 대한
회귀 테스트도 함께 실행한다.

## 2단계 확장 경계

2단계는 1단계를 실제 사용한 뒤 별도 설계와 구현으로 진행한다. 그 단계에서는 여러 이미지,
본문 내 삽입 위치, 이미지 크기와 캡션 편집, 키보드 및 클립보드 기반 인라인 편집 형식,
기존 한 장 첨부의 자동 변환 규칙을 결정한다. 1단계에서는 이 기능을 노출하거나 숨은 HTML을
저장하지 않는다.
