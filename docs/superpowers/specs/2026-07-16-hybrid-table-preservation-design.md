# 하이브리드 표 원형 보존 설계

## 목표

PDF와 스캔 문서에 포함된 표를 파싱, 문제은행 저장, 문제 편집, 문제 추출 및 DOCX 출력 전 과정에서 보존한다. 모든 표는 원본 영역 이미지와 편집 가능한 구조 데이터를 함께 저장하며, DOCX에서는 자동·원본 이미지·편집 가능한 표 출력 방식을 지원한다.

## 현재 구조의 한계

현재 PDF 파서는 PyMuPDF가 인식한 일부 텍스트 표를 행·열 문자열로만 추출한다. 표와 문항의 연결은 셀 문자열이 발문에 포함됐는지로 판정하고, DB의 `question_format_json`에 단순 `rows`만 저장한다. 이 과정에서 다음 정보가 손실된다.

- 셀 병합, 열 너비, 행 높이
- 정렬, 테두리, 음영
- 표 안의 수식, 도형, 이미지, 회전 글자
- 발문이나 선지 안에서 표가 있던 정확한 위치
- OCR 문서의 표 구조

문제 편집기는 발문을 저장할 때 기존 `tables`를 보존하지 않고 `spans`만 다시 만들기 때문에 표 정보가 사라질 수 있다. DOCX exporter는 모든 표를 단순 `Table Grid`로 발문 뒤에 추가하므로 원래 위치와 서식을 재현하지 못한다.

## 선택한 접근법

하이브리드 방식을 사용한다.

1. 모든 표의 원본 영역을 고해상도 이미지로 보존한다.
2. 인식 가능한 표는 행·열·셀·병합·서식 정보를 구조 데이터로 함께 저장한다.
3. DOCX 출력은 구조 복원 신뢰도와 복잡도에 따라 Word 표 또는 원본 이미지를 자동 선택한다.
4. 사용자는 문서 전체 또는 개별 표의 출력 방식을 직접 선택할 수 있다.

검토에서 제외한 방식은 다음과 같다.

- 이미지만 저장: 원형은 보존하지만 편집 가능한 Word 표를 만들 수 없다.
- 구조 데이터만 저장: OCR 오류, 수식, 복잡한 병합이 있는 표의 원형을 복구할 수 없다.

## 데이터 모델

DB 스키마를 추가로 분리하지 않고 기존 `questions.question_format_json`과 `question_choices.choice_format_json`을 버전 2로 확장한다. `tables` 항목에 원본과 구조를 함께 저장하며 기존 `rows` 필드는 하위 호환을 위해 유지한다.

```json
{
  "schema_version": 2,
  "spans": [],
  "tables": [
    {
      "id": "table-1",
      "anchor": {
        "offset": 18,
        "before_context": "다음 표를 보고",
        "after_context": "옳은 것은"
      },
      "rows": [["구분", "값"], ["A", "10"]],
      "cells": [
        {
          "row": 0,
          "col": 0,
          "text": "구분",
          "row_span": 1,
          "col_span": 1,
          "horizontal_alignment": "center",
          "vertical_alignment": "center"
        }
      ],
      "column_widths": [0.5, 0.5],
      "row_heights": [0.34, 0.33],
      "borders": [],
      "source": {
        "image_path": "data/table_images/table-1.png",
        "source_pdf_relative_path": "source.pdf",
        "page": 3,
        "bbox": [120, 240, 820, 510],
        "sha256": "source-image-sha256"
      },
      "confidence": {
        "score": 0.94,
        "reasons": ["native_grid", "all_words_assigned"]
      },
      "complexity": {
        "has_formula": false,
        "has_embedded_image": false,
        "has_rotated_text": false,
        "has_complex_merge": false
      },
      "render_mode": "auto",
      "recommended_render": "native"
    }
  ]
}
```

동일한 구조를 발문과 선지에 사용한다. 표 이미지 경로는 문제은행 ZIP과 포터블 앱 생성 시 함께 복사할 수 있는 상대 경로로 저장한다.

## 파싱 파이프라인

### 원본 이미지 보존

표 후보가 발견되면 구조 인식 결과와 관계없이 원본 PDF 영역을 고해상도 PNG로 잘라 저장한다. 이미지에는 원본 PDF 상대 경로, 페이지, 좌표, 해시를 연결한다. 구조 인식이 실패하더라도 이 이미지는 표 원형의 최종 폴백으로 남는다.

### 텍스트 PDF

PyMuPDF의 표·셀 좌표와 페이지 drawing 정보를 결합한다. 셀 사각형, 내부 선의 유무, 텍스트 좌표를 사용해 행·열, 셀 병합, 폭·높이, 정렬과 테두리를 복원한다.

### OCR PDF

렌더링된 페이지에서 가로·세로 선을 검출해 격자를 구성하고, OCR 단어 좌표를 각 셀에 할당한다. 내부 선이 없는 영역은 인접 셀 병합 후보로 처리한다. 테두리 없는 표는 단어의 X/Y 정렬과 반복 간격으로 구조를 추정하되 낮은 신뢰도를 부여한다.

### 복잡도와 신뢰도

수식, 도형, 이미지, 회전 글자, 대각선, 불규칙 병합은 복잡도 플래그로 기록한다. 구조 신뢰도는 다음 근거를 종합한다.

- 격자 검출 완전성
- OCR 단어의 셀 할당률
- 행·열 경계의 일관성
- 비어 있거나 중복된 셀 비율
- 원본 텍스트와 셀 텍스트의 일치도

### 문제와 표 연결

문자열 포함 여부 대신 페이지 좌표를 사용한다. 문제 영역, 발문 영역, 각 선지 영역과 표 bbox의 포함·교차 관계로 표 소유자를 결정한다. 표에서 유래한 평문은 발문이나 선지에서 제거하고 동일 위치의 저장 텍스트 삽입 offset과 앞·뒤 문맥을 anchor로 남겨 중복 출력을 막는다.

## 편집 화면

표 위치에는 다음 기능을 가진 표 카드를 표시한다.

- 원본 보기: 파싱 당시의 원본 표 이미지
- 구조 보기: 인식된 행·열 표
- 표 편집: 셀 텍스트, 셀 병합, 정렬 수정
- 출력 방식: 자동, 원본 이미지, 편집 가능한 표
- 원본 이미지 교체와 복원

편집 저장 시 `spans`만 다시 만들지 않고 기존 표 payload를 함께 병합한다. 발문 수정으로 숫자 offset이 무효가 되면 앞·뒤 문맥을 사용해 위치를 다시 찾는다. 문맥도 찾지 못하면 표를 삭제하지 않고 발문 또는 선지 끝에 배치하고 사용자에게 경고한다.

## DOCX 출력

모의고사 출력 화면에 문서 전체 표 출력 방식 선택을 추가한다.

- 자동: 신뢰도 0.90 이상이고 복잡도 플래그가 없으면 Word 표, 그 외에는 원본 이미지
- 원본 이미지: 원본 표 이미지를 그대로 삽입
- 편집 가능한 표: 구조 데이터를 Word 표로 재구성

개별 표에 명시적으로 저장된 `render_mode`가 `auto`가 아니면 문서 전체 기본값보다 우선한다.

Word 표는 셀 병합, 폭·높이, 정렬, 테두리와 음영을 적용한다. 표가 현재 2단의 한 열보다 넓으면 해당 표 앞뒤에 연속 구역 나누기를 추가해 표 부분만 1단으로 출력한다. 표 블록은 anchor 순서에 따라 발문과 선지 사이의 원래 위치에 삽입하며 평문과 중복하지 않는다.

구조 출력이 실패하면 원본 이미지로, 원본 이미지가 없으면 유효한 구조 표로 자동 폴백한다. 둘 다 없고 `rows`만 남아 있으면 단순 Word 표를 만들고 내보내기 경고에 기록한다. 표 문제 때문에 전체 DOCX 생성이 중단되지는 않는다.

## 기존 DB 보강

- 기존 `tables: [{"rows": [["구분", "값"]]}]`는 읽을 때 메모리에서 버전 2 형태로 정규화한다.
- 원본 PDF가 있는 문항은 전체 재파싱해 표 이미지와 구조를 보강한다.
- 원본이 없는 문항은 현재 `rows`를 유지하며 존재하지 않는 원본 이미지를 만들지 않는다.
- Mounted 문제은행의 쓰기 대상과 읽기 전용 정책을 그대로 적용한다.
- 실제 DB 변경 전에 staging DB에서 검증하고 원본 DB를 백업한 후 트랜잭션 단위로 반영한다.

## 자산 포터블 처리

문제은행 ZIP과 포터블 앱 생성기는 다음 위치의 표 이미지 참조를 모두 수집한다.

- `question_format_json.tables[*].source.image_path`
- `choice_format_json.tables[*].source.image_path`

자산을 포터블 이미지 디렉터리에 복사하고 JSON 내부 경로를 상대 경로로 다시 작성한다. 누락된 표 이미지는 원본 문제 이미지 누락과 같은 수준으로 검증 보고서에 기록한다.

## 품질 게이트

DB 반영 전에 다음 조건을 검사한다.

- 표 이미지 존재 및 SHA-256 일치
- bbox가 원본 페이지 안에 존재
- 행·열·셀 좌표와 병합 범위가 유효
- 한 OCR 단어가 여러 셀에 중복 할당되지 않음
- 표 평문이 발문이나 선지에 중복되지 않음
- 표가 정확히 하나의 문제 또는 선지에 연결됨
- anchor offset이 저장 텍스트 범위 안에 있거나 앞·뒤 문맥으로 복원 가능
- 구조 신뢰도가 낮은 표에 원본 이미지가 존재

품질이 낮은 구조 데이터는 DB 반영을 무조건 차단하지 않는다. 원본 이미지가 유효하면 `recommended_render: image`로 저장한다. 원본 이미지까지 없을 때만 오류 대상으로 격리한다.

## 테스트 전략

단위 및 회귀 테스트에 다음 fixture를 추가한다.

- 텍스트 PDF의 일반 표와 병합 표
- OCR 스캔의 격자 표
- 테두리 없는 표
- 수식·도형·회전 글자가 포함된 표
- 발문 중간과 선지 안의 표
- 넓은 표의 DOCX 1단 전환
- 자동·원본 이미지·편집 가능한 표 출력
- 편집 후 원본 이미지, 구조 데이터와 anchor 유지
- 기존 버전 1 표 데이터의 호환
- Mounted 문제은행의 읽기·쓰기 경계
- ZIP과 포터블 앱의 표 이미지 경로 재작성

생성된 DOCX는 OpenXML을 다시 읽어 표 위치, 셀 병합, 폭, 이미지, 구역 단 수를 검사한다. 이미지 출력은 원본 crop의 해시와 DOCX media payload를 비교한다. 전체 테스트와 실제 DB staging 회귀를 모두 통과한 뒤에만 DB를 교체한다.

## 완료 조건

- 파싱된 모든 표 후보에 원본 crop이 있거나 명시적인 누락 사유가 있다.
- 구조 데이터가 있는 표는 DB 저장·조회·편집 후 동일하게 유지된다.
- DOCX의 세 출력 모드가 발문과 선지 표에 모두 적용된다.
- 자동 모드는 복잡한 표를 원본 이미지로, 단순 고신뢰도 표를 Word 표로 출력한다.
- 기존 문제은행과 Mounted 문제은행의 비표 문항에 회귀가 없다.
