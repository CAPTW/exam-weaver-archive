# 파싱 문항 자동 품질 게이트 설계

## 목표

PDF/OCR 파싱 결과를 운영 DB에 곧바로 적재하지 않고, 문항 단위 검수와 DB 단위 검수를 연속으로 통과한 데이터만 Mounted DB로 승격한다. 확실하게 교정할 수 있는 표기 오류는 공통 정규화 단계에서 자동 보정하고, 의미가 불명확한 문항은 원문을 보존한 채 격리하여 잘못된 자동 교정을 막는다.

## 선택한 구조

파서와 staging DB가 같은 텍스트 품질 규칙을 공유하는 2단계 게이트를 적용한다.

1. **파서 후보 게이트**
   - 기존 구조·정답·선지 수 검증에 OCR 노이즈, 깨진 단위, 괄호 불균형, 비정상 특수기호 검사를 추가한다.
   - 명백한 표기 정규화 뒤에도 문제가 남은 문항은 `RejectedOfflineQuestion`으로 격리한다.
   - 격리 사유, 발문, 선지, 페이지, 신뢰도와 진단 정보를 staging DB에 저장한다.
2. **staging 최종 게이트**
   - 적재가 끝난 DB 전체를 `QuestionValidator`의 문항 품질 규칙으로 다시 스캔한다.
   - OCR/구조 무결성 오류가 하나라도 있으면 검증 보고서를 실패로 만들고 Mounted DB 교체를 차단한다.
   - 경고성 메타데이터와 태그 정책은 OCR 승격 차단과 분리하여 오탐을 줄인다.
3. **원자적 승격**
   - 기존 staging 검증, 백업, 원자적 파일 교체 흐름을 유지한다.
   - 새 품질 게이트를 통과한 DB만 Mounted DB가 될 수 있다.

## 공통 품질 규칙

OCR 노이즈 정규식, 깨진 단위 정규식, 괄호 균형 검사는 parser 계층의 독립 모듈로 이동한다. 파서 후보 검증과 DB 검증이 이 모듈을 함께 사용하여 서로 다른 판정이 생기지 않게 한다. 기존 DB 수리 스크립트도 동일한 괄호 판정기를 래핑해 사용한다.

자동 보정은 NFKC 정규화, 공백·줄바꿈 정리, 이미 검증된 화살표/기호 정규화처럼 의미가 변하지 않는 규칙에 한정한다. 문맥을 추측해야 하는 OCR 오류는 자동 수정하지 않고 격리한다.

## 격리 및 보고

staging DB에 문항별 격리 테이블을 추가한다. 각 행에는 소스 상대 경로, 시험/과목/연도/회차, 문항 번호와 페이지, 사유 코드 목록, 발문·선지 원문, 신뢰도, 파서 진단을 기록한다.

빌드 산출물에는 기존 검증 JSON/CSV와 함께 격리 JSON을 추가한다. 최종 DB 스캔에서 발견된 차단 항목도 검증 보고서에 문항 ID, 사유 코드, 요약으로 포함한다. 따라서 어떤 문항이 왜 배치되지 않았는지 재현할 수 있다.

## 오탐 방지

Mounted DB 승격을 차단하는 코드는 빈 발문, OCR placeholder, 의심 문자열, OCR 노이즈, 깨진 단위, 괄호 불균형, 잘못된 연도·회차·문항 번호·정답 상태·선지 구조, 누락 이미지처럼 문항 무결성에 직접 관련된 항목으로 제한한다. 태그 누락과 같은 운영 메타데이터 경고는 보고하되 이 게이트에서 승격을 막지 않는다.

## 검증 전략

- 화살표 꼬리, OCR 노이즈, 깨진 단위, 괄호 불균형이 있는 파싱 후보가 격리되는 단위 테스트
- 정상 특수기호와 올바른 괄호가 오탐 없이 통과하는 단위 테스트
- 의심 문항이 포함된 staging DB가 최종 품질 게이트에서 실패하는 통합 테스트
- 격리 상세 행과 JSON 보고서가 생성되는 통합 테스트
- 깨끗한 staging DB의 승격 가능성을 보장하는 회귀 테스트
- 전체 pytest 스위트 실행

## 배포 및 호환성

기존 공개 importer는 공통 `parse_offline_question_pdf` 경로를 사용하므로 파서 후보 게이트가 자동 적용된다. staging 스키마 추가는 새 빌드 DB에만 적용하며 운영 DB의 기존 문항 스키마는 변경하지 않는다. 기존 Mounted DB 교체 API는 그대로 유지되고, 검증 실패 시 기존 운영 DB는 손대지 않는다.
